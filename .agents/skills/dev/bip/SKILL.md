# dev:bip — Build in Public ツイート生成

git log と会話コンテキストから、X（旧Twitter）に投稿できる日本語ツイート3案を生成します。
「何を作ったか」を報告するのではなく、**読んだ人が反応したくなる角度**で切り取ることがこのスキルのゴールです。

## なぜ普通の進捗報告はエンゲージメントが取れないか

「〇〇機能を追加しました」「リファクタしました」は事実ですが、読者に**「自分ごと」感**がありません。
エンゲージメントが高いツイートは以下のいずれかを持っています：

- **意外性・逆説**: 「当然こうだと思ってたけど、違った」
- **緊張感・ドラマ**: 「詰まってた / やらかした / やっと解決した」
- **具体的な数字**: 「3時間で / 8スキルを / 20件のコミット」
- **普遍的な気づき**: 「これ、○○している人全員に当てはまるかも」
- **問いかけ**: 読んだ人が「あるある」「自分も知りたい」と感じる終わり方

## 実行手順

### Step 1: コンテキスト収集

以下を実行してください：

```bash
# 直近20件のコミット（日時付き）
git log --format="%aI %s" -20

# プロジェクト概要（存在する方を使用）
head -40 CLAUDE.md 2>/dev/null || head -40 README.md 2>/dev/null
```

### Step 2: 「ツイートの角度」を見つける

単純に「何を作ったか」をまとめるのではなく、**読者が反応する角度**を探してください。

git log と会話コンテキストから以下を分析します：

| 探す観点 | 問い |
|----------|------|
| **驚き・逆説** | 「やってみたら想定と違ったことはあるか？」「当たり前と思ってたことが覆されたか？」 |
| **詰まり・解決** | 「困っていたことがあったか？それをどう解決したか？」 |
| **before/after** | 「作業前と後で何がどう変わったか？」数字があればなお良い |
| **普遍的な学び** | 「この気づきは他の開発者にも当てはまるか？」 |
| **なぜ作っているか** | 「この作業の背後にある動機・目的は何か？」 |

**コンテキストが不明な場合**: 以下の一言をテキストで問いかけてください：
> 「今日のツイートで特に伝えたいことを教えていただけますか？（例：詰まったこと、気づき、達成したことなど）」

### Step 3: ツイート3案を生成

**制約**:
- 日本語のみ
- **280字以内**（全角・半角ともに1文字 = 1カウント）
- ハッシュタグなし

**3案で異なる構文を使ってください**（同じ事実を、異なるフレームで切り取る）：

#### 構文リファレンス

以下から3案それぞれに最もフィットするものを選んでください。

**逆説・意外性型**
```
「○○すればいい」と思っていたけど、全然違った。
〔具体的に何が違ったか〕
〔教訓・次のアクション〕
```

**before/after 型**
```
〔作業前の状態〕→〔作業後の状態〕
〔数字や具体的な変化〕
〔なぜそうしたか、または気づき〕
```

**発見・気づき型**
```
〔作業中に気づいたこと〕
〔具体的なエピソード1行〕
〔普遍化・読者へのメッセージ〕
```

**詰まり・突破型**
```
〔何で詰まっていたか〕
〔どう突破したか〕
〔同じ状況の人へ〕
```

**動機・なぜ作るか型**
```
〔作っているものの名前・機能〕
〔なぜこれを作ろうと思ったか、個人的なエピソード〕
〔これを作ることで何が変わるか〕
```

#### 避けるべきパターン

- ❌ 「〇〇機能を追加しました」だけで終わる（事実の羅列）
- ❌ 「着々と進んでいます」（抽象的すぎる）
- ❌ 「みなさんもぜひ」（宣伝くさい）
- ❌ 誇張・過剰な自己アピール（「驚愕の」「革命的な」）

#### 品質チェック

各案を出力する前に、以下を自問してください：
- 「この1文だけ読んだら、続きを読みたくなるか？」
- 「読んだ人は『あるある』か『知らなかった』と感じるか？」
- 「自分が他の人のタイムラインでこれを見たら、反応するか？」

### Step 4: 出力

以下のフォーマットで出力してください：

---

**案1（〔使った構文名〕）**
〈ツイート本文〉
（〇〇字）

---

**案2（〔使った構文名〕）**
〈ツイート本文〉
（〇〇字）

---

**案3（〔使った構文名〕）**
〈ツイート本文〉
（〇〇字）

---

文字数は正確にカウントしてください。構文名は「逆説型」「before/after型」などを括弧内に記載します。
