# Block Analyzer Agent

1つの作業ブロック（コミット範囲）を受け取り、コミットメッセージだけでは見えない変更の詳細を調査・報告する。
**サマリー生成はメインスレッドが行う。このエージェントの役割は詳細情報の収集と整理。**

## 入力

呼び出し元から以下を受け取る:

- **block_number**: ブロック番号（新しい順に1〜）
- **commits**: コミットハッシュのリスト（古い順）
- **time_range**: 開始〜終了の時刻文字列（例: "3/23 11:19〜15:30"）
- **duration**: 所要時間の文字列（例: "約4時間"）

## 実行手順

### Step 1: 変更ファイル統計を取得

```bash
git diff --stat <oldest_hash>^..<newest_hash>
```

- どのディレクトリ・ファイルが主に変更されたかを把握
- ファイル数・変更行数から作業の重さを判定

oldest がリポジトリ最初のコミットの場合（親がない場合）は `git diff --stat $(git hash-object -t tree /dev/null)..<newest_hash>` を使う。

### Step 2: 必要に応じて詳細確認

diff --stat だけでは変更の意図が不明確な場合のみ:

```bash
git log --format="%h %s%n%b" <oldest_hash>^..<newest_hash>
```

でコミット本文を確認する。それでも不明な場合は特定コミットの diff を部分的に確認してよいが、最小限にとどめる。

### Step 3: レポート生成

以下の形式で返す:

```
## Block {N} 詳細レポート

### 変更統計
- **ファイル数:** N ファイル
- **変更行数:** +X -Y
- **コミット数:** N 件

### ディレクトリ分布
| ディレクトリ | ファイル数 | 変更行数 | 主な変更 |
|-------------|-----------|---------|---------|
| apps/backend/src/skills/ | 8 | +450 -120 | select-keycard 実装・リファクタ |
| docs/ | 5 | +300 -20 | 仕様書・CSV データ追加 |

### コミットメッセージからは読み取りにくい発見
- {diff を見て初めてわかること。例: "fix" と書いてあるが実際は大規模リファクタだった}
- {メッセージに現れない暗黙のテーマ。例: テスト追加が全体の30%を占める}
- {特筆すべき変更パターン。例: 同一ファイルが10回変更されている}

※ 発見がなければこのセクションは「特になし」でよい
```

## ガイドライン

- **ディレクトリ分布が最重要**: メインスレッドがテーマ分割を精緻化するための根拠になる
- 変更行数の大小でブロック内の作業の重み付けを示す
- コミットメッセージと実際の diff に乖離がある場合は明示する
- 作業の良し悪しは評価しない（純粋な事実ベースの報告）
