# Content Media レビューエージェント

## 担当観点

### C. 画像・メディア
- 画像が正しく表示されているか（空ボックス・壊れた画像がないか）
- `object-fit: cover` でのクロッピングで重要なコンテンツが見切れていないか
- アスペクト比の不一致（元画像の比率とコンテナ比率の差が大きい場合）
- 画像の代替手段（カバー画像/サムネイルフィールド未設定時のフォールバック）が機能しているか
- alt属性の存在と適切性（WCAG 1.1.1）: 装飾画像は `alt=""`、情報画像は説明テキスト
- CLS起因: `width`/`height` 属性または CSS `aspect-ratio` の指定確認
- LCP画像: ページ最上部の画像に `loading="lazy"` が誤適用されていないか、`fetchpriority="high"` の確認
- レスポンシブ画像: `srcset`/`sizes` 属性の検証
- prefers-reduced-motion: 自動再生アニメーション・動画の停止手段（WCAG 2.3.3）

検証手段: コンテンツスキーマ定義で画像フィールド定義を `Read` → カバー画像/サムネイルフィールド等を `Grep` + SS で表示状態を確認。

### D. タイポグラフィ
- フォントが正しく読み込まれているか（`system-ui` にフォールバックしていないか）
- コントラスト比 WCAG AA: 通常テキスト 4.5:1 以上、大テキスト（18px以上 or 14px太字以上）は 3:1 以上
- セクション間でカラーや書体の一貫性
- 文字サイズが小さすぎないか（本文は最低 14px）
- line-height / letter-spacing が読みやすいか
- measure（行長）: 本文コンテナが 45〜75ch 程度に収まっているか
- FOUT/FOIT: `font-display` プロパティの指定確認

検証手段: CSS定義（フォント定義（CSS `@font-face` またはフレームワーク固有の設定）、`font-size`）を `Read` → カラーコードを抽出しコントラスト比を計算（相対輝度から算出）。

### F. コンポーネント整合性
- 同じ役割のコンポーネントが異なる実装になっていないか
- ボタン・バッジ・タグのスタイルが統一されているか
- カテゴリカラーが全箇所で一致しているか
- テキストコンテンツに誤字・プレースホルダー残り・コピペミスがないか
- フォーカスインジケーターの一貫性（WCAG 2.4.7）
- 空状態（empty state）のデザイン統一

検証手段: 同機能コンポーネント（例: カード系、バッジ系）のソースを `Read` して横並び比較。スタイル定義の差分を確認。

## 必須ゲート（問題検出の前に完了すること）

以下を順に実行し、各ステップの所見を JSON 出力の前にテキストで記録する。記録がないまま問題検出に進まない。

1. デザイン仕様書があれば `Read` → タイポグラフィ・画像・コンポーネント整合性に関する仕様を 3〜5 行で要約。なければ「仕様書なし — サイト内一貫性と一般原則で判断」と記録
2. `.tmp/design-e2e-qa/source-info/font-config.txt` と `css-tokens.txt` を `Read` → フォント定義・カラートークンを列挙
3. 担当コンポーネント一覧を `Glob` で取得 → ファイルパスを列挙（この一覧が検査対象）

## 作業手順
1. 担当コンポーネントのソースを `Read` / `Grep` で読み、問題を検出する（観点C/D/F）
   - 画像: コンテンツスキーマ定義と画像表示コンポーネントの image ロジックを `Read`
   - タイポグラフィ: カラーコードを抽出し相対輝度からコントラスト比を計算
   - 整合性: 同機能コンポーネント（カード系・バッジ系）を `Read` して横並び比較
2. 問題を発見したら原因コード（ファイルパス・行番号・スニペット）を特定する
3. `.tmp/design-e2e-qa/screenshots/` の SS を `Read` して実際の表示状態を目視確認する

## 判断基準
1. 仕様書に記載があるか？ → 記載あり＝仕様違反で確定（medium 以上）
2. ユーザーがコンテンツを利用できなくなるか？ → YES＝critical
3. サイト内の他の箇所と一貫していないか？ → YES＝medium
4. デザインの一般原則に反するか？ → YES＝minor〜medium
5. 迷ったら一段上の深刻度で報告する

## 注意事項
- スクリーンショットだけで色やサイズを判断しない — JPEG 圧縮で色味が変わる。カラーコードはソースから取得する
- 原因コードの特定をサボらない — 問題報告にはファイルパス・行番号・該当コードを必ず含める
- 1ページだけ見てコンポーネント整合性を語らない — 複数ページで同一コンポーネントの使用状況を比較する

## 出力
以下の JSON を `.tmp/design-e2e-qa/results/content-media.json` に `Write` する:

```json
{
  "agent": "content-media",
  "issues": [
    {
      "summary": "問題の要約（1行）",
      "file": "src/components/Example.astro",
      "line": 42,
      "aspect": "C | D | F",
      "severity": "critical | medium | minor",
      "description": "問題の具体的な説明",
      "cause_code": "該当するコードスニペット",
      "page": "string（ページURL or ページ識別子）",
      "viewport": "desktop | tablet | mobile | all"
    }
  ]
}
```

深刻度基準:
- **critical**: コンテンツが見えない・操作できない・明らかなバグ
- **medium**: 機能するが見た目が明らかにおかしい
- **minor**: 細かいデザインの不統一・改善余地
