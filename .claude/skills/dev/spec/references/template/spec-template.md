# 仕様書テンプレート

仕様書 `docs/PLAN/{YYMMDD}_{slug}.md` は以下の構成で作成する。

---

## 必須セクション

```markdown
# {タイトル}

## 実行プロトコル（この仕様書の使い方）

この仕様書に基づいて作業するエージェントは、以下のプロトコルに従うこと。

### Todo 実行の手順

各 Todo は必ず以下の **3 ステップを順番に** 実行する。ステップを飛ばしてはならない。

```
Step 1 — IMPL    実装を行う
Step 2 — REVIEW  Task(sonnet) で todo-review.md を実行し、結果を本ファイルの記入欄に貼り付ける
Step 3 — FIX     FAIL がある場合のみ修正（最大3ラウンド）。全 PASS なら不要
```

**REVIEW の実行方法**: Task ツール（model: sonnet）で `.claude/skills/dev/spec/tasks/todo-review.md` を Read し、そのプロンプトに従って実行する。引数として本仕様書のパスと対象 Review ID を渡す。

### Gate 通過条件

Gate 内の全 Todo について:
1. IMPL が完了していること
2. REVIEW 結果記入欄に **レビュー結果テーブルが記入済み** であること（空欄は不可）
3. 全 REVIEW の総合判定が **PASS** であること

### 注意事項

- **IMPL 完了後、REVIEW を実行せずに次の Todo に進んではならない**
- REVIEW 結果記入欄が空の Todo がある状態で Gate 通過としてはならない
- この仕様書は自己完結している。「参照すべきファイル」に記載されたファイルを読んでから着手すること

## 概要
```

## タスクリストの実行方式タグ

各タスクにオプションで実行方式を付与できる:

```
`[実行方式:モデル:サブエージェント]`
```

### 記法ルール

| 記法                 | 意味                                           |
| -------------------- | ---------------------------------------------- |
| （タグなし）         | メインスレッドで実行（デフォルト）             |
| `[BG:haiku:Explore]` | バックグラウンドで haiku、Explore エージェント |
| `[PARALLEL]`         | 次の複数タスクを並列実行                       |

### モデル選択基準

| モデル | 用途                                   |
| ------ | -------------------------------------- |
| haiku  | 単純な検索・ファイル読み込み・定型作業 |
| sonnet | 中程度の複雑さの実装・分析             |
| opus   | 複雑な設計・重要な判断                 |

### サブエージェントタイプ

| タイプ          | 用途             |
| --------------- | ---------------- |
| Explore         | コードベース探索 |
| general-purpose | 汎用タスク       |
| code-simplifier | コード整理       |

## ワークフロー分類

各タスクの `ワークフロー` フィールドには以下のいずれかを指定する:

| ワークフロー | 対象                                               | 判定基準                                       |
| ------------ | -------------------------------------------------- | ---------------------------------------------- |
| **TDD**      | ビジネスロジック、バリデーション、計算、データ変換 | 入出力が明確に定義でき、アサーションで検証可能 |
| **E2E**      | UI コンポーネント、レイアウト、ユーザー操作        | 視覚的確認または UX/UI 判断が必要              |
| **TASK**     | 設定ファイル、環境構築、ドキュメント               | テスト不要、コマンド実行結果で検証可能         |

判定フロー:

```
入出力が明確に定義できる？
    ├─ YES → TDD
    └─ NO → 視覚的確認が必要？
              ├─ YES → E2E
              └─ NO → TASK
```
