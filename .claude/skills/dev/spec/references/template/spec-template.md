## タスクリスト

### 実装→レビュー→修正ループの方針

各 Todo は以下のパターンで実行する:

1. **IMPL**: 実装エージェントが Todo を実行
2. **REVIEW**: 質的レビューエージェント（sonnet）が 6 観点で自律的にレビュー
3. **FIX**: 指摘があれば実装エージェントが修正（最大 3 ラウンド）
4. **GATE**: レビュー通過 → 次の Todo へ

**REVIEW の実行方法**: Task ツール（model: sonnet）で `.claude/skills/dev/spec/tasks/todo-review.md` を Read し、そのプロンプトに従って実行する。引数として本仕様書のパスと対象 Review ID を渡す。
**レビュー 6 観点**: ストーリー適合性 / API・SDK 準拠 / ベストプラクティス / リスク・前提の検証 / セキュリティ / 保守性・可読性

### 依存関係図

[Gate 単位で依存関係を図示]
```

Gate A: {フェーズ名}
├── Todo 1 → Review A1
└── Todo 2 → Review A2

Gate B: {フェーズ名}（Gate A 完了後）
└── Todo 3 → Review B3

```

### Gate N: {フェーズ名}

#### Todo N: {タイトル}

- [ ] **IMPL**
  - **対象**: {ファイルパス}
  - **内容**: {何を — 具体的なアクション}
  - **実装詳細**: {どうやって — 既存パターン参照、コード例}
  - **ワークフロー**: TDD / E2E / TASK
  - **依存**: {先行タスク — なければ「なし」}

- [ ] **Review N** — 質的レビューエージェント（sonnet）が実行
  - **レビュー方式**: 6 観点（ストーリー適合性 / API・SDK 準拠 / ベストプラクティス / リスク・前提の検証 / セキュリティ / 保守性・可読性）で自律的に評価。該当しない観点は N/A
  - **判定**: 全観点 PASS（または N/A）→ Gate N 通過。FAIL があれば指摘を返し FIX ラウンドへ

## タスクリストの実行方式タグ

各タスクにオプションで実行方式を付与できる:

```
`[実行方式:モデル:サブエージェント]`
```

### 記法ルール

| 記法                 | 意味                                           |
| -------------------- | ---------------------------------------------- |
| （タグなし）         | メインスレッドで実行（デフォルト）             |
| `[BG:haiku:Explore]` | バックグラウンドで haiku、Explore エージェント |
| `[PARALLEL]`         | 次の複数タスクを並列実行                       |

### モデル選択基準

| モデル | 用途                                   |
| ------ | -------------------------------------- |
| haiku  | 単純な検索・ファイル読み込み・定型作業 |
| sonnet | 中程度の複雑さの実装・分析             |
| opus   | 複雑な設計・重要な判断                 |

### サブエージェントタイプ

| タイプ          | 用途             |
| --------------- | ---------------- |
| Explore         | コードベース探索 |
| general-purpose | 汎用タスク       |
| code-simplifier | コード整理       |

## ワークフロー分類

各タスクの `ワークフロー` フィールドには以下のいずれかを指定する:

| ワークフロー | 対象                                               | 判定基準                                       |
| ------------ | -------------------------------------------------- | ---------------------------------------------- |
| **TDD**      | ビジネスロジック、バリデーション、計算、データ変換 | 入出力が明確に定義でき、アサーションで検証可能 |
| **E2E**      | UI コンポーネント、レイアウト、ユーザー操作        | 視覚的確認または UX/UI 判断が必要              |
| **TASK**     | 設定ファイル、環境構築、ドキュメント               | テスト不要、コマンド実行結果で検証可能         |

判定フロー:

```
入出力が明確に定義できる？
    ├─ YES → TDD
    └─ NO → 視覚的確認が必要？
              ├─ YES → E2E
              └─ NO → TASK
```
