# 実装仕様書作成（dev:spec）

## ゴール

**別の作業エージェントに渡せる自己完結した実装仕様書**を作成する。

「自己完結」とは:
- 仕様書の中に、作業に必要なすべてのコンテキスト（設計判断・コード構造・参照資料）が含まれている
- 作業エージェントが外部リポジトリや他のスキルの知識なしに実装を完遂できる
- ファイルパスはすべてコードベース内で解決可能（外部パスが紛れ込んでいない）

## 出力

| ファイル | 内容 |
|----------|------|
| `docs/PLAN/{YYMMDD}_{slug}.md` | 仕様書本体 |
| `docs/PLAN/{YYMMDD}_{slug}/references/` | 参照資料（外部ファイルのコピー、API 仕様等の新規リファレンス。必要な場合のみ） |

---

## ★ 実行手順（この順序を厳守）

### Step 0: コンテキスト収集

1. **会話コンテキストを分析**し、仕様書の対象タスクを特定する
   - 直前の会話で議論されている機能・設計・作業内容があればそれが対象
   - 特定できない場合のみ AskUserQuestion で候補を提示して確認
2. **CLAUDE.md** を Read してプロジェクト構成・技術スタックを把握する
3. **関連ドキュメント・コード**を探索する（Glob, Grep, Read）

### Step 1: 要件の深さ判定

以下の 4 項目を判定する:

| # | 判定項目 | 基準 |
|---|----------|------|
| 1 | 技術選定 | 主要な技術的意思決定が確定しているか？ |
| 2 | スコープ | 変更範囲が明確か？何を含み何を含まないか？ |
| 3 | データフロー | コンポーネント間のデータ受け渡しが定義されているか？ |
| 4 | エッジケース | エラーハンドリング・境界条件の方針があるか？ |

- **2 項目以上が未確定** → Step 1a（深掘り）へ
- **1 項目以下が未確定** → Step 2 へスキップ

### Step 1a: 要件の深掘り

未確定項目を解消するために `/dev:dig` を実行する。

**ゲート**: Step 1 の 4 項目がすべて確定するまで Step 2 に進まない。

### Step 2: タスク分解

仕様書に載せるタスクリストを作成するために `/dev:decomposition` を実行する。

### Step 3: 仕様書ドラフト作成

1. 日付を取得:
   ```bash
   date +%y%m%d
   ```

2. AskUserQuestion で仕様書の slug を確定する

3. `references/template/spec-template.md` を Read し、その構成に従って `docs/PLAN/{YYMMDD}_{slug}.md` に Write する

### Step 4: 外部参照資料の同梱

仕様書内で参照しているファイルをすべてチェックし、**作業エージェントがアクセスできないもの**を `docs/PLAN/{YYMMDD}_{slug}/references/` にコピーまたは作成する。

#### 判定基準

| 条件 | アクション |
|------|-----------|
| 他リポジトリのファイルを参照 | `cp` でコピー |
| `.reference/` サブモジュール内を参照 | `cp` でコピー |
| 外部 API ドキュメントを参照 | 要点を markdown で新規作成 |
| 会話中に出てきた API 仕様・プロトコル定義・データ構造 | リファレンス markdown を新規作成 |
| 特定のライブラリ/SDK の使い方に依存する設計 | 必要なパターン・API を markdown で新規作成 |

#### コピー後の必須処理

1. **コピーしたファイル内のパス修正**:
   - 元リポジトリの相対パス（例: `docs/PLAN/some-path/file.yml`）が残っていないか Grep で確認
   - 該当箇所に「元リポジトリ {repo名} 内のファイル。本リポジトリには存在しない」と注記
2. **仕様書本体の参照パス更新**:
   - 外部絶対パスを `docs/PLAN/{YYMMDD}_{slug}/references/` 内のパスに書き換え
3. **最終確認**:
   - 作業エージェントが「このリポジトリ内にある」と誤解するパス記述がないか通読

### Step 5: レビュー

→ **Task（sonnet）** に委譲（`tasks/plan-review.md` を Read してプロンプトとして使用）

レビューエージェントに渡す情報:
- 仕様書のファイルパス
- CLAUDE.md のパス（存在する場合）

レビュー観点:
1. **タスク粒度** — 各タスクが 1-2 時間で完了可能か
2. **依存関係** — 正しい順序か、循環依存はないか
3. **自己完結性** — 仕様書だけで作業エージェントが実装できるか
4. **参照の完全性** — 外部参照がすべてコピー済みか、パスが正しいか
5. **漏れ** — セットアップ・エラーハンドリング・テストタスクの不足
6. **リスク** — 残存リスクの緩和策は十分か

### Step 5a: レビュー修正ループ

レビュー結果が **NEEDS_REVISION** の場合:

1. 指摘事項をユーザーに提示
2. AskUserQuestion で修正方針を確認:
   - 指摘に従い自動修正する
   - ユーザーが手動で調整する
   - 指摘を無視して続行する
3. 修正を反映して Step 5 に戻る

**最大 2 回のループ**。3 回目は NEEDS_REVISION でもユーザー判断に委ねる。

### Step 6: ユーザー最終確認

仕様書の完成を報告し、AskUserQuestion で次のアクションを確認:

- **コミットする** → `/dev:simple-add` でコミット
- **修正が必要** → 指示に従い修正
- **実装に進む** → 作業エージェントへの引き渡し方を案内

## 完了条件

- [ ] `docs/PLAN/{YYMMDD}_{slug}.md` が存在する
- [ ] 仕様書内の全参照ファイルがコードベース内パスで解決可能
- [ ] 外部参照は `references/` にコピー済み、パスも修正済み
- [ ] レビューが完了（APPROVED or ユーザー判断で続行）
- [ ] ユーザーが承認済み
