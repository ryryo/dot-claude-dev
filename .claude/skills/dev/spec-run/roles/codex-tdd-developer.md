# Codex TDD Developer — プロンプトテンプレート

Codex モードで [TDD] タスクを `codex exec` に委任する際のプロンプトテンプレート。
Claude オーケストレーターがこのテンプレートの `{変数}` を埋めて stdin 経由で渡す。

## 実行コマンド

```bash
cat /tmp/codex-tdd-prompt.md | codex exec -m gpt-5.4 --sandbox workspace-write --full-auto --ephemeral - 2>/dev/null
```

- `--ephemeral`: セッション再利用による前タスクの文脈汚染を防止

## フォールバック

- codex exec が失敗した場合: エラー情報を含めてプロンプトを改善し、最大3回リトライ
- 3回失敗: Claude が `roles/tdd-developer.md` のロール定義で直接実装

## プロンプトテンプレート

Claude オーケストレーターは以下のテンプレートを埋めて一時ファイルに書き出す。

```markdown
## タスク

以下の仕様に基づいて TDD で実装してください。

### 手順

1. テストファイルを作成し、期待する振る舞いを記述する（RED）
2. テストを実行し、失敗することを確認する
3. テストを通す最小限の実装を書く（GREEN）
4. テストが通ることを確認する
5. コードの品質を改善する（REFACTOR）
6. テストが引き続き通ることを確認する

### 原則

- テストを先に書かずに実装を始めない
- テストを変更してグリーンにしない
- YAGNI — 仕様に書かれていないものは作らない
- Baby steps — 小さなステップで進める

## 仕様

{仕様書の Todo IMPL 内容}

## コンテキスト

### プロジェクト情報
{プロジェクトの技術スタック、ディレクトリ構造の要約}

### 関連コード
{参照すべきファイルの内容を要約 — 10,000文字以内に収める}

## 制約

- 確認や質問は不要。テスト作成から実装完了まで自主的に進めること
- git commit は行わない（オーケストレーターが行う）
- 既存コードの構造・命名規則・スタイルに合わせる
- 仕様に記載されていない機能は追加しない
{仕様書の設計決定事項があれば追記}

## 完了条件

- すべてのテストがパスすること
- リファクタリング後もテストがパスすること
```

## テンプレート変数の埋め方

| 変数 | 取得元 | 注意事項 |
|------|--------|----------|
| `{仕様書の Todo IMPL 内容}` | 仕様書の対象 Todo の IMPL セクション | そのまま引用 |
| `{プロジェクトの技術スタック...}` | CLAUDE.md + package.json 等 | 簡潔に要約 |
| `{参照すべきファイルの内容を要約}` | 仕様書の「参照すべきファイル」を Read | 10,000文字以内に要約。超える場合は関数シグネチャ・型定義に絞る |
| `{仕様書の設計決定事項...}` | 仕様書の設計決定事項セクション | あれば制約に追加 |

## コンテキストサイズの目安

| サイズ | 判定 | 対応 |
|--------|------|------|
| < 2,000文字 | 最適 | そのまま使用 |
| 2,000-10,000文字 | 許容 | 不要な部分を削除して使用 |
| > 10,000文字 | 要圧縮 | 関数シグネチャ・型定義・インターフェースに絞る |
