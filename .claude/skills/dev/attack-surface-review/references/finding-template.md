# 攻撃面レビュー レポートテンプレート

このテンプレートは、調査サブエージェントが**そのままメインセッションに返す完成レポート形式**である。
メモや断片ではなく、この構造で返すこと。

## レビュー概要
- **レポート種別:** `static-review` または `dynamic-validation`
- **対象:** アプリ名、ディレクトリ、レビュー対象範囲
- **重点観点:** 今回重点的に見た攻撃面
- **確認した入口:** HTTPルート、Webhook、管理画面、バッチ、外部連携など
- **実行した既存チェック:** 実行コマンドと結果の要約
- **制約 / 未確認範囲:** 実環境依存、外部サービス未接続、権限不足など

## 所見サマリー
- **High以上件数:** N
- **Medium件数:** N
- **総評:** 1段落で記述

## [Severity] タイトル
- **対象箇所:** `path/to/file`（関数名/クラス名まで特定する）
- **攻撃ベクター:** 攻撃者が実行する具体的な操作を1文で記述
- **前提条件:** 権限、ネットワーク位置、被害者操作の要否
- **悪用シナリオ:** 2〜5ステップで再現可能な流れを記述
- **影響:** 機密性/完全性/可用性への影響を明示
- **根拠:** 該当コードの挙動・設定値・検証結果
- **検証ステータス:** `コード上で成立可能` / `動的に再現済み` / `今回の範囲では未成立`
- **再現手順:** 動的検証をした場合は手順を記述。未実施なら `未実施`
- **Payload / リクエスト例:** 実際に投げた値、HTTPリクエスト、入力例。なければ `なし`
- **観測結果:** レスポンス、ログ、画面挙動、テスト結果。なければ `なし`
- **未成立理由 / 阻害要因:** 再現できなかった場合のみ記述
- **確信度:** High / Medium / Low
- **対策方向:** 実装コードではなく、方針を1〜3行で記述

## 所見なしの場合
- `確認範囲では再現可能な脆弱性は確認できなかった`
- 未確認範囲・制約を明記する
- 次に見るべき攻撃面があれば1〜3件だけ挙げる

## レポート末尾
- **全体リスク評価:** 1段落
- **優先対策トップ3:** 箇条書き
- **次段への入力:** 動的検証に回す候補、または dev:spec に回す候補を箇条書き
- **エスカレーション判定:** Medium以上が1件でもあれば dev:spec へ誘導
