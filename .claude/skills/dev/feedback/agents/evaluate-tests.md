# evaluate-tests

## 役割

TDDで作成されたテストを評価し、保持/簡素化/削除の判断材料を提供する。

## 推奨モデル

**haiku** - 単純な分類タスク

## 入力

- TDDで作成されたテストファイル一覧
- 各テストの内容

## 処理

各テストについて以下を判断:

| 判断基準 | アクション |
|---------|-----------|
| **長期的な価値がある** | テスト資産として保持・維持 |
| **一時的な価値のみ** | テストを簡素化、または削除候補 |

## 評価ポイント

- **回帰テストとしての価値**: 将来の変更で壊れたことを検知できるか
- **メンテナンスコスト**: 実装変更のたびにテスト修正が必要か
- **ドキュメントとしての価値**: テストが仕様を伝えているか

## プロンプト

```
TDDで作成されたテストを評価してください。

## テストファイル
{test_files}

## 評価基準

各テストを以下の観点で分類:

1. **保持推奨**: 回帰テストとして価値がある、仕様のドキュメントになっている
2. **簡素化候補**: 価値はあるが冗長、テストケースを統合可能
3. **削除候補**: 実装詳細に依存、メンテナンスコストが高い、一時的なデバッグ用

## 出力形式

- 保持推奨: [テスト名一覧と理由]
- 簡素化候補: [テスト名一覧と理由]
- 削除候補: [テスト名一覧と理由]
```

## 出力

テスト分類結果（保持/簡素化/削除の一覧と理由）
