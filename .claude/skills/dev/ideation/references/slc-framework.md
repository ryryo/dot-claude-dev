# SLC（Simple, Lovable, Complete）フレームワーク

## MVPとの違い

| | MVP | SLC |
|---|---|---|
| 目的 | 学習のための最小限 | 愛されるための最小限 |
| 品質 | 妥協あり | 妥協なし |
| ユーザー体験 | 不完全でもOK | 完全な体験を提供 |
| スコープ | 機能を削る | 機能を削るが残した部分は磨く |

## SLCの3条件

### Simple（シンプル）
- 1つのコアジョブに集中
- 余分な機能を排除
- 理解に説明書が不要

### Lovable（愛される）
- 使って嬉しい体験
- 細部への配慮
- 「また使いたい」と思える

### Complete（完全）
- 提供する機能は完全に動作
- エッジケースも処理
- 中途半端な機能がない

## スコープ決定プロセス

```
1. コアジョブを1つ特定
2. そのジョブに必要な最小機能を列挙
3. 各機能を「必須 / あると嬉しい / 不要」に分類
4. 「必須」のみでSLCを構成
5. 各必須機能の品質基準を定義
```

## 検証質問

- この機能なしでコアジョブは達成できるか？ → Yes なら削除
- この機能は中途半端になっていないか？ → Yes なら磨くか削除
- ユーザーは初回利用で価値を感じるか？
- 競合と比べて明確な差別化があるか？
