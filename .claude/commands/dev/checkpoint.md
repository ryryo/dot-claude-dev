---
description: コンテキスト圧迫時にメモリ保存→/clear→作業再開を支援する。タスク分解済みならTodoツール登録用フォーマットで保存。
---

コンテキストが圧迫されてきたので、現在の作業状態をメモリに保存し、/clear 後に作業を再開できるようにしてください。

## 手順

### Step 1: 現在のメモリファイルを確認

`memory/MEMORY.md` と `memory/tasks/` 配下のファイルを Read で確認し、既存の内容を把握する。

### Step 2: 保存すべき情報を整理

以下をすべて網羅すること:

1. **タスクの目的と背景**（なぜこの作業をしているか）
2. **決定事項**（dig/質問で確定した選択肢・方針）
3. **タスク分解**（Todo リストがあればそのまま保持。Phase/依存関係も含む）
4. **現在の進捗**（どの Todo まで完了したか、未着手はどれか）
5. **主要ファイルパス**（変更対象・参照先の絶対パス一覧）
6. **技術的な注意事項**（ハマりポイント、仕様上の制約、環境情報）
7. **未解決の問題**（ブロッカー、保留事項）

### Step 3: メモリファイルに書き込む

- `memory/MEMORY.md`: トップレベルのタスク参照（200行以内に収める）
- `memory/tasks/<task-slug>.md`: 詳細なタスク情報

**タスク分解が存在する場合**、各 Todo を以下のフォーマットで記述する（再開時に TaskCreate でそのまま登録可能）:

```markdown
## Todos（TaskCreate 登録用）

### Todo 1: <タイトル>

- **状態**: pending | in_progress | completed
- **依存**: なし | Todo X
- **説明**: <そのTodoだけで実行できる十分な情報。ファイルパス・変更内容・検証方法を含む>

### Todo 2: <タイトル>

...
```

各 Todo の「説明」は、そのまま TaskCreate の description に使えるレベルの具体性を持たせること:

- 変更対象ファイルの絶対パスと行番号（わかる場合）
- 具体的な変更内容（何を追加/削除/変更するか）
- 検証方法（コマンドやアサーション）
- 参考パターンがある場合はそのファイルパス

### Step 4: 再開手順を表示

作業の種類に応じてコピペ文を使い分ける:

**A. 仕様書が存在する場合**（`/dev:spec` 後など、`docs/PLAN/` 配下に spec ファイルがある）:

```
メモリ保存完了。以下の手順で再開してください:

1. /clear でコンテキストをクリア
2. 以下をコピペして送信:

memory/tasks/<task-slug>.md を読んで作業を再開してください。
その後、/dev:spec-run を Codex モードで実行してください。
```

**B. 仕様書がなく Todo リストのみの場合**（実装途中での中断など）:

```
メモリ保存完了。以下の手順で再開してください:

1. /clear でコンテキストをクリア
2. 以下をコピペして送信:

memory/tasks/<task-slug>.md を読んで作業を再開してください。
Todos セクションの pending タスクを TaskCreate で登録し、依存関係に従って順番に実行してください。
```

### Step 5: 保存内容サマリーを出力（必須）

再開手順に続けて、**何を保存したか**を以下のフォーマットで必ず出力する。ユーザーが `/clear` 前に確認するための最後のレビュー材料。

````markdown
## 保存内容サマリー

### 書き込んだファイル

| ファイル | 操作 | 内容 |
|----------|------|------|
| `memory/tasks/<slug>.md` | 新規作成 / 更新 | 決定事項 N 項目・主要パス N 件・Todo N 件（全 pending） |
| `memory/MEMORY.md` | Active Tasks 追記 | 1 行エントリ |

### 作業状態のスナップショット

- **最終コミット**: `<hash>` `<subject>`（仕様書など成果物がコミット済みの場合）
- **未ステージ変更**: `<path>`（温存する理由を1行で） / または「なし」
- **Todo 進捗**: `<completed>/<total>`（pending N / in_progress N / completed N）

### 再開準備チェック

- [x] Todo description は TaskCreate にそのまま渡せる具体性（ファイルパス・変更内容・検証方法）
- [x] ファイルパスは絶対パス
- [x] 各 Todo に検証コマンドまたはアサーションを記載
- [x] 参考実装位置（仕様書のセクション名等）を明示
````

**フォーマットルール**:

- 各セクションで該当項目がなければその行を省略してよい（例: 未ステージ変更がなければ行ごと削除）
- テーブルは最低1行、最大5行程度に収める。長くなるなら「ほか N 件」でまとめる
- チェック欄は**実際に満たしている項目のみ** `[x]`。満たしていなければ `[ ]` のまま残して改善点として可視化
- 絵文字・装飾は使わない（`✅`・`📝` 等は禁止）
- **再開手順のコードブロックの直後**に出力する。順序: 再開手順 → 保存内容サマリー

### 注意事項

- 既存のメモリファイルがある場合は上書きではなく適切に更新する
- completed のタスクファイルは簡潔にまとめて保持（詳細は削除してよい）
- 1つのタスクファイルが長くなりすぎる場合は分割を検討
