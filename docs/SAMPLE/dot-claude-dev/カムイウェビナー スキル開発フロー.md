# カムイウェビナー スキル開発フロー

このフローの核心は、**「AIに自律的に動いてもらいつつ、人間が管理・改善できるポイントを明確にする」**という点にあり、徹底的な**トークン節約**と**再現性の確保**が設計思想となっています。

***

# Claude Code「スキル」活用型 開発フロー完全ガイド

## 1. 開発フローの基本思想：再現性の追求
この手法の目的は、アプリそのものを作ること以上に、**「誰がやっても高品質で同じ成果物が出る開発の仕組み（再現性）」**を構築することにあります。

### 従来手法からの脱却（失敗からの学び）
*   **以前の手法**: 200個以上のスキルを作成し、細かく使い分けていた。
*   **問題点**: トークン消費が激しい（メタ情報だけで消費される）、AIが混乱する、管理が煩雑。
*   **現在の手法**: **「抽象度の高い3つのスキル」**と**「具体性の高い13のタスク仕様書」**の組み合わせに移行。
    *   **効果**: トークン消費を約半分（20〜25% → 10〜15%）に削減し、実行速度と品質が向上。
    *   **設定**: トークン節約のため、MCP（Model Context Protocol）機能はオフにする。

---

## 2. コア・コンポーネントの構成

開発の基盤となるのは、以下の3種類の「スキル」と「タスク仕様書」です。

### A. 3つの主要スキル（The "Big 3"）
200個あったスキルを以下の3つ（+α）に集約しています。

1.  **タスク仕様書作成用スキル**: システム仕様書を読み込み、実装機能に応じた具体的な手順書（タスク仕様書）を生成する。
2.  **スキルクリエイター（Skill Creator）**:
    *   新しいスキルの作成だけでなく、**「システム仕様書の更新（フィードバック）」**に使用される最重要スキル。
    *   フォーマットを強制し、仕様書に不要なコードが混入するのを防ぐ「型チェック（Lint）」のような役割を果たす。
3.  **システム仕様書作成用スキル**: ゼロから仕様書を作る、あるいは大きく改変する際に使用。

### B. スキルの内部構造
スキルは、`SKILL.MD`をハブとして機能させ、記述を最小限に抑えています。
*   **構成要素**: エージェント、スクリプト、リファレンス、設定ファイル。
*   **役割**: `SKILL.MD`がこれらを呼び出し、ワークフローを自動化する。

### C. タスク仕様書（The 13 Phases）
巨大な1つのプロンプト（8万文字レベル）ではなく、**13個のフェーズごとのファイル**に分割して管理します。

**【主なフェーズ構成】**
1.  **計画 (Plan)**: 全体の進行計画を立てる。
2.  **設計 (Design)**: 実装詳細を設計する。
3.  **テスト駆動開発 (TDD)**: テストコードを先に作成し、実装する。
4.  **テスト補充**: カバレッジ80%以上を目指してテストを追加する。
5.  **リファクタリング**: コードを整理し最適化する。
6.  **品質保証 (QA)**: 動作確認と品質チェック。
7.  **ドキュメント作成/フィードバック**: 開発内容の文書化。
8.  **プルリクエスト (PR)**: 最終成果物の提出。
    *(※他、合計13フェーズに細分化)*

## 3. 具体的な開発実行サイクル

開発は以下の**「循環型フロー（フィードバックループ）」**で進められます。

### Step 1: タスク仕様書の生成
*   **入力**: システム仕様書（全体設計図）。
*   **アクション**: 「タスク仕様書作成スキル」を使用し、今回実装する機能に特化したタスク仕様書を生成する。

### Step 2: 分割タスクの実行（バトンリレー）
生成されたタスク仕様書（13ファイル）を順番に実行します。
*   **脳のフレッシュ化**: フェーズごとにファイルを分けることで、コンテキスト（AIの記憶容量）を圧迫せず、常に高い精度を維持。
*   **中断と再開**: 各フェーズが独立しているため、エラーやトークン切れで止まっても、そのフェーズから容易に再開可能。

### Step 3: レポート出力（必須工程）
*   **ルール**: 各フェーズ（設計、TDDなど）の終了時に、必ず**「レポート」**を出力させる。
*   **保存先**: `output/`フォルダ配下に格納。
*   **目的**:
    *   AIの暴走（無限ループ）を防ぐ。
    *   人間が進捗と品質をレビューできるようにする。
    *   次のフェーズへの引き継ぎ資料とする。

### Step 4: システム仕様書へのフィードバック
開発完了後、得られた知見や変更点を**「システム仕様書」**に書き戻します。
*   **アクション**: **「スキルクリエイター」**を使用して更新を行う。
*   **重要性**: 単にテキストを追記するのではなく、スキルを経由することで「仕様書のフォーマット」を守り、品質を維持する（GitHubのIssueテンプレートを守らせる感覚）。

## 4. このフローで得られるメリット

| メリット | 詳細 |
| :--- | :--- |
| **コスト削減** | スキル読み込みを最小限にし、タスクを分割することでトークン消費を半減。 |
| **高い再現性** | 細分化されたタスクとレポート確認により、誰が実行しても同じ品質のコード・ドキュメントが生成される。 |
| **品質の安定** | 「レポート出力」がストッパーとなり、AIが勝手に突き進んでコードを破壊するリスクを排除。 |
| **持続的な改善** | 開発するたびに仕様書が洗練され、スキル自体も「スキルクリエイター」で改善できる自己進化型の仕組み。 |

## 5. まとめ：AI時代の「指揮者」になるために
このフローは、AIにすべてを丸投げするのではなく、**「仕様書（全体図）→タスク（詳細指示）→実装→フィードバック（全体図の更新）」**というサイクルを、人間がスキームとして管理するものです。

大子氏は、AIの進化に合わせてツールが変わっても（Claude Code以外になっても）、この**「フィードバックループを回して改善し続けるサイクル」**こそが重要であると結論づけています。
