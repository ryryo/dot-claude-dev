# ストーリー駆動AI開発

## 概要

AI時代の開発において、従来の仕様駆動開発よりもストーリー駆動開発のほうが有効である。これは本質的にはスクラムの考え方をAI開発に適用したものだが、AI特有の認知負荷の問題に対処する実践的なアプローチとなる。

## 仕様駆動開発の問題点

### 認知負荷の高さ
- 詳細な仕様書を作ってAIに渡すと、人間もAIも認知負荷が高すぎて破綻する
- 長大な仕様書では、AIも人間も「何を今やるべきか」を見失う
- 仕様の全体を把握しながら開発を進めるのは非現実的

### 柔軟性の欠如
- 仕様変更時の影響範囲が大きい
- 仕様書全体の整合性を保つコストが高い
- AIが仕様書全体を考慮した判断をするのは困難

## ストーリー駆動開発のアプローチ

### 基本概念
- **ユーザーストーリーの塊**でドキュメントを小さく群として管理
- それをもとにAIも人間も開発を進める
- 仕様駆動よりも柔軟性が高くうまくいく

### ユーザーストーリーと仮説の関係
- ユーザーストーリーは**仮説の塊**である
- 「仮説駆動開発」と「ストーリー駆動開発」は本質的に同じ意味
- ただし、見た目のオブジェクト的な名前では「ストーリー駆動」のほうが適切

### 注意点
- 「ストーリー駆動」という名前だと、重厚長大なストーリーを書いてくる危険性がある
- 「いやそれほぼSpecやん」という状態になりがち
- あくまで小さく、焦点を絞ったストーリーであることが重要

## ストーリーと仕様書の関係

### 開発フロー
```
ユーザーストーリー → 開発 → 仕様書に集約
```

### 仕様書の新しい役割

**開発を駆動するものではない**
- 仕様書は「仕様駆動開発」のためのものではない
- むしろ「仕様駆動保守」のためのもの

**コンテキストの圧縮**
- ストーリーの羅列だとあとから辛い
- ストーリーを繋げて意味の塊として整理
- 保守運用のために記憶しやすいように作る
- ロングコンテキストを耐え抜くための工夫

**人間でも同様**
- スクラムでもそうだが、ストーリーを中心にやっていくのは良い
- しかし仕様というドキュメントはドキュメントで大事
- AI時代も仕様を作ったほうがいい

## 用語の整理

### 正確な意味
- **ストーリー駆動開発** = ただのスクラム
- **スペック駆動開発** = ただのウォーターフォール
- 正確には「ストーリー駆動AI開発」「スペック駆動AI開発」という文脈での話

### 前から存在する用語の問題
- 言葉通りなら既存の開発手法と同じになってしまう
- AI開発という文脈を明示することで区別する必要がある

## 実践的な運用

### 開発時
1. ユーザーストーリーを小さな単位で作成
2. 各ストーリーをドキュメント化して群として管理
3. AIも人間も、必要なストーリーを参照しながら開発
4. ストーリー単位で実装を完了させる

### 保守時
1. 完了したストーリーを仕様書に統合
2. 関連するストーリーを意味のある塊としてまとめる
3. 仕様書は全体像把握と詳細確認のために使用
4. 新規開発時は再びストーリーベースで進める

## なぜストーリー駆動が良いのか

### 認知負荷の低減
- 「今、誰のために、何を作るか」が明確
- 一度に考える範囲が限定される
- AIも人間も理解しやすい粒度

### 柔軟性の確保
- 仕様変更時も該当ストーリーを修正すれば済む
- 影響範囲が局所的
- 優先順位の変更に対応しやすい

### AIとの相性
- AIは「文脈のある依頼」のほうが理解しやすい
- 長大な仕様書よりも焦点を絞ったストーリーのほうが適切な出力を得られる
- 必要な情報を必要なタイミングで参照できる

## まとめ

### 開発の駆動力
- **ストーリー**: 開発を駆動する燃料
- 小さく、焦点を絞った単位で進める
- 人間もAIも認知負荷を下げる

### 保守の支援
- **仕様書**: 保守・運用のためのコンテキスト圧縮
- ストーリーを統合した全体像
- 後から見返したときの理解を助ける

### AI時代の開発スタイル
- 開発: ユーザーストーリーを中心に小さく進める
- 保守: ストーリーを統合した仕様書で全体像を把握
- AIの特性(長文理解はできるが、長文から適切に行動を選ぶのは苦手)に合わせた実践的なアプローチ
