# Discovery / Decision Sprint

PLAN 作成前に行う深掘り工程。結果は `docs/PLAN/{YYMMDD}_{slug}/references/discovery.md` に記録する。

## Goal

現在の要件や初期プランに潜む隠れた前提、未発見のリスク、未決定事項を明らかにし、解消可能な不明点を Gate 契約へ反映できる状態にする。

## Principles

- 広く浅くではなく、重要な糸口を深く掘る。
- 前提そのものに疑問を呈する。
- 暗黙の決定事項を明示する。
- ユーザーがまだ考えていない可能性が高い論点を優先する。
- 解消可能な不明点を「残存リスク」に棚上げしない。

## Phase 1: Context Collection

質問する前に、利用可能な情報を読む。

- 現在の会話で示された要件。
- 既存 PLAN がある場合は `spec.md` と `tasks.json`。
- README、AGENTS、CLAUDE、package manager、主要 config。
- 関連ドキュメント、PRD、仕様書、既存 issue。
- 関連コード、テスト、既存実装、参照実装。

特定するもの:

- 明示された目標。
- 明示された制約。
- 暗黙の前提。
- 欠落している主要トピック。
- 実装判断に必要だが未確認の事実。

このフェーズでは、確認できる事実を先に確認する。コードやドキュメントで解けることをユーザー質問にしない。

## Phase 2: Assumption Mapping

質問を作る前に、前提をカテゴリ別に並べ、誤っていた場合の影響で優先順位を付ける。

| カテゴリ | 例 |
| --- | --- |
| 実現可能性 | 選んだ技術で実装できる、性能が足りる |
| ユーザー行動 | ユーザーはこの導線で操作する、設定を理解できる |
| スコープ | 今回含めるもの、含めないもの |
| 依存関係 | 外部サービス、SDK、API、認証、契約状態 |
| スケジュール | 期限、段階リリース、移行期間 |
| アーキテクチャ | 既存構造で支えられる、既存 state と衝突しない |
| データ/状態 | 永続化、キャッシュ、同期、権限、削除/復元 |
| 運用 | 監視、ロールバック、保守担当、障害時対応 |

各前提に `risk: high | medium | low` を付け、high から調査する。

## Phase 3: Deep Interview

Plan Mode の質問機能で、1ラウンドあたり 2〜3 問だけ聞く。各質問は 2〜4 個の具体的な選択肢を持つ。

質問ルール:

- 自由記述を基本にしない。
- 各選択肢に短いメリット/デメリットを付ける。
- 「その他」は質問ツール側が追加するため、選択肢に含めない。
- 既存コードのパターンに沿った選択肢を優先する。
- 1つの重要トピックは、可能なら2段階以上掘る。

質問カテゴリ:

- 前提: 「プランは X を前提としている。これは正しいか？」
- トレードオフ: 「X を選ぶと Y を犠牲にする。どちらを優先するか？」
- スケール: 「現在は N 件で動く。10N ではどうするか？」
- 障害モード: 「X が失敗したとき、何をユーザーに見せるか？」
- ユーザーシナリオ: 「権限不足、空データ、初回利用、既存利用者でどう振る舞うか？」
- 依存関係: 「外部 API / 契約 / 認証が使えないときどうするか？」
- セキュリティ/プライバシー: 「誰が何にアクセスできるか？」
- 保守性: 「ローンチ後に誰がどこを保守するか？」
- マイグレーション/ロールバック: 「現状態から安全に移行し、戻すにはどうするか？」
- 優先順位衝突: 「品質、速度、互換性、UX のどれを優先するか？」

各回答後:

1. 新しく明らかになった前提を追加する。
2. 最もリスクの高い糸口をフォローアップする。
3. 新トピックへ移る前に、主要トピックを十分に掘る。
4. 未探索カテゴリを記録する。

## Phase 4: Apply Findings

各ラウンド後、発見事項を PLAN の材料として整理する。

```markdown
## 発見事項（ラウンドN）

### 挑戦した前提

| 前提 | 判明したこと | 影響 | 決定 |
| --- | --- | --- | --- |

### 決定事項

| トピック | 決定内容 | 根拠 | リスクレベル |
| --- | --- | --- | --- |

### 新たに浮上した質問

- ...
```

反映先:

- 確定した決定は `spec.md` の `設計決定事項`。
- 実行判断に必要な制約は `tasks.json.gates[].constraints`。
- 検証すべき条件は `acceptanceCriteria`。
- 実行前準備は `preflight`。
- 解消不能な外部要因だけ `残存リスク`。

## Phase 5: Completeness Check

次を満たすまで Gate 設計へ進まない。

- 高リスク前提が明示的に対処されている。
- 主要トピックは最低2段階の深さで確認されている。
- Phase 4 の「新たに浮上した質問」が残っていない。
- トレードオフが明示され、決定されている。
- クリティカルパスの障害モードが扱われている。
- 決定事項が `spec.md` / `tasks.json` へ反映できる形になっている。
- 解消可能な不明点が残存リスクに残っていない。

未達項目があれば Phase 3 に戻る。

## Stop Conditions

次の場合は質問を増やさず Gate 設計へ進む。

- 残りの不明点が計画段階で解消不能な外部要因だけ。
- 追加質問の答えが Gate 契約や設計判断を変えない。
- ユーザーが明示的に現時点の決定で進めると選んだ。

次の場合は Gate 設計へ進まない。

- 高リスク前提が未確認。
- AC が書けない。
- 検証手段が決まっていない。
- 破壊禁止対象やスコープ境界が曖昧。
- 主要な依存関係の利用可否が未確認。
