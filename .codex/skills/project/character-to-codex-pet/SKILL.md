# 既存キャラクターからCodexペットを作る

既存画像をいきなりスプライトシートへ変換せず、まずピクセルアートの基準画像を1枚だけ作る。ユーザーが明示的に承認した画像を正準画像として固定し、その画像に接地した状態差分だけを生成する。

## 必須ルール

- 最初に `${CODEX_HOME:-$HOME/.codex}/skills/.system/imagegen/SKILL.md` を最後まで読み、通常は組み込みの `image_gen` を使う。
- Codexペットの準備、配置ガイド、フレーム抽出、検証、合成には `hatch-pet` を使う。インストール済みパスを優先し、なければ `${CODEX_HOME:-$HOME/.codex}/vendor_imports/skills/skills/.curated/hatch-pet` を確認する。使用前に対象の `SKILL.md` を最後まで読み、現在の引数と成果物を正とする。
- 既存画像がローカルファイルだけの場合は、生成前に `view_image` で表示してから編集対象として渡す。
- 元画像について、キャラクターの使用許可がある前提を確認する。ロゴや読める文字は、ユーザーが明示的に残すよう依頼しない限り除外する。
- 基準画像の承認前にアニメーション行を生成しない。
- 「いいです」「OK」「これで進めて」など、対象画像を特定できる明示的な承認だけを採用する。
- 承認済み画像は上書きせず、`references/canonical-base.png` として固定する。
- 差分生成では正準画像と状態別レイアウトガイドを毎回添付する。テキストだけで差分を生成しない。
- 生成画像に最終アトラスの厳密な寸法を期待しない。切り出し、透明化、合成、検証は `hatch-pet` の決定的スクリプトに任せる。
- 失敗時は最小単位の行だけを再生成する。正準画像や問題のない行を作り直さない。
- 透明背景は組み込み生成で単色クロマキーを使い、ローカル処理でアルファ化する。真の透過が必要な場合だけ、`imagegen` の規則に従ってCLIフォールバックの許可を得る。

## 進捗表示

開始時に次のチェックリストを提示し、常に1項目だけを進行中にする。

1. 元画像とキャラクター条件を確認する。
2. ピクセルアートの基準画像を作る。
3. 基準画像の承認を得る。
4. 9状態のアニメーション差分を作る。
5. ペットを合成・検証・配置する。

## ワークフロー

### 1. 入力を確認する

次を画像またはユーザーの説明から確定する。判断できる項目は質問せず推定し、結果を短く提示する。

- 参照画像の絶対パス
- ペット名、ID、1文の説明
- 必ず維持する特徴：顔、体型、配色、服装、模様、小物
- 削除してよい情報：背景、文字、ロゴ、影、装飾
- 希望するピクセル密度、輪郭、色数。未指定なら小さなUIで読みやすい中密度のピクセルアートにする
- 作業用runディレクトリ

参照画像が複数ある場合は役割を明示する。主画像を編集対象、残りを顔・服装・配色などの補助参照として扱う。

### 2. runを準備する

`hatch-pet` の `prepare_pet_run.py` を使い、参照画像、`pixel` スタイル、キャラクター固定条件を渡してrunを準備する。生成された `pet_request.json`、`imagegen-jobs.json`、基準画像プロンプト、状態別プロンプト、レイアウトガイドを正とする。

準備後、基準画像ジョブ以外は実行しない。

### 3. 基準画像を1枚だけ生成する

既存画像を編集対象とする `style-transfer` として `imagegen` を使う。プロンプトは [prompts-ja.md](references/prompts-ja.md) の「基準画像」を読み、元画像から推定した固定条件を追加する。

次を満たす画像だけを候補にする。

- 元キャラクターと同一人物・同一個体に見える
- 全身が中央にあり、身体が切れていない
- 本物のピクセルクラスタとして読め、滑らかなイラストへ格子を重ねただけに見えない
- 影、床、背景、文字、透かし、浮遊記号がない
- キャラクター内でクロマキー色を使っていない
- `192x208` 相当まで縮小しても顔と輪郭を識別できる

候補をユーザーに表示し、維持した特徴と懸念点を簡潔に伝える。ここで必ず停止して承認または修正指示を待つ。

### 4. 基準画像を修正する

承認されなかった場合は、1回につき1つの主要変更だけを加える。前候補を編集対象として使い、変更しない特徴をプロンプト内で繰り返す。新しい候補は別ファイルとして保存し、過去候補を上書きしない。

ユーザーが承認したら、その画像をrunのdecoded出力へ記録し、`references/canonical-base.png` として固定する。承認日時と元ファイルをmanifestに記録する。

### 5. アニメーション差分を生成する

9状態を次の順序で作る。

1. `idle`
2. `running-right`
3. `running-left`
4. `waving`
5. `jumping`
6. `failed`
7. `waiting`
8. `running`
9. `review`

各行は別の `imagegen` 呼び出しにする。必ず正準画像と、その行のレイアウトガイドを役割付きで添付する。プロンプトはrun内の状態別プロンプトを正とし、[prompts-ja.md](references/prompts-ja.md) の「差分生成」の同一性固定を補助的に適用する。

`running-right` のQA後、文字、ロゴ、左右固有の模様、小物の向き、光源の意味が反転で壊れない場合だけ、`hatch-pet` の専用スクリプトで `running-left` を派生する。判断理由を記録する。それ以外は個別生成する。

差分生成をサブエージェントへ委任する場合は、ユーザーが許可していなければ開始前に確認する。許可された場合も同時生成は最大2行とし、各担当には1行だけを任せる。親はmanifest更新、正準画像管理、合成、修復判断を担当する。

### 6. 合成とQAを行う

全行が揃ったら `hatch-pet` のスクリプトで次を順に実行する。

1. 行画像からフレームを抽出する。
2. フレーム構成と不要成分を検査する。
3. `1536x1872`、8列×9行、1セル`192x208`のアトラスへ合成する。
4. WebPを検証する。
5. コンタクトシートと状態別プレビューを作る。
6. [qa-ja.md](references/qa-ja.md) に従って目視確認する。

自動検証が成功しても、同一性の変化や不自然な動きがあれば不合格にする。抽出処理が原因のサイズ跳ねは、画像を再生成する前に `stable-slots` による再抽出を検討する。

### 7. 修復する

不合格行だけを、正準画像、元の行プロンプト、レイアウトガイド、具体的な失敗理由を添えて再生成する。修復後は抽出、検査、合成、検証、プレビュー確認を再実行する。High相当の問題がなくなるまで繰り返す。

### 8. 配置する

合格後に次の2ファイルを同時に配置する。

```text
${CODEX_HOME:-$HOME/.codex}/pets/<pet-id>/
├── pet.json
└── spritesheet.webp
```

`pet.json` には `id`、`displayName`、`description`、`spritesheetPath` を含める。配置後、Codex Appの「設定 → 外観 → ペット／アバター」から選択するよう案内する。表示されない場合はCodex Appを再起動する。

## 完了条件

- ユーザーが承認した正準画像が保存されている。
- 全9状態が正準画像と同じキャラクター、ピクセル密度、配色、輪郭、服装、小物に見える。
- `final/validation.json` にエラーがない。
- コンタクトシートと全プレビューを目視確認している。
- 未使用セルが完全に透明である。
- `pet.json` と `spritesheet.webp` が同じペットディレクトリに配置されている。
- 最終報告に正準画像、コンタクトシート、検証結果、配置先の絶対パスを記載している。
